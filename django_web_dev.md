# Django & Web Security AMA

### 1. What is Clickjacking?

Clickjacking  is a malicious technique where an attacker tricks a user into clicking on a hidden or transparent element on a webpage. The attacker overlays a malicious invisible layer (often using a transparent `<iframe>`) over a legitimate page. When the user thinks they are clicking on a button like a video play button or a game, they are actually clicking an invisible link that performs actions on another site, such as making a purchase, deleting an account, or modifying permissions.

### 2. Why do we use `render()` in Django?

In Django views, `render()` is a shortcut function used to generate a full web page response. It combines a specified HTML template with a context dictionary containing dynamic data, processes it through Django's template engine, and automatically wraps the resulting HTML string into a proper `HttpResponse` object that can be sent back to the client's browser. It simplifies web page delivery into a single line of code.

### 3. What is a CSRF token?

A **CSRF (Cross-Site Request Forgery) token** is a unique, secret, and unpredictable value generated by a server-side application to protect state-changing requests (like submitting a form, changing a password, or making a transfer). It ensures that the submission originated from the legitimate web application and not from a malicious third-party site executing a hidden request on behalf of an authenticated user.

### 4. Do we use `DEBUG = True` in production?

**Absolutely not.** In production environments, `DEBUG` must always be set to `False`. When `DEBUG = True`, any unhandled exception or crash will expose a highly detailed diagnostic page (the Django yellow screen of death) to the public. This traceback reveals sensitive configuration settings, environment variables, database names, internal file structures, and source code excerpts, creating an extreme security vulnerability. It also degrades application performance.

### 5. Why do we use `collectstatic`?

In Django, individual applications often store their static assets (like CSS, JavaScript files, and theme images) inside their respective local directories. The `python manage.py collectstatic` command loops through all registered apps, gathers these scattered files, and copies them into a single, centralized directory specified by `STATIC_ROOT`. This unified folder makes it easy for dedicated production web servers (such as Nginx or Apache) to serve static assets efficiently without touching Django.

### 6. Difference between `get()` and `filter()` in Django?

* **`get()`**: Used to retrieve a single, specific object from the database matching the criteria. It evaluates immediately and throws an exception if the item is not found (`DoesNotExist`) or if more than one item matches (`MultipleObjectsReturned`).
* **`filter()`**: Used to look up items that match the criteria and always returns a `QuerySet` (a list-like collection of objects). If no matches are found, it returns an empty `QuerySet` instead of crashing, making it safe for handling multiple or zero results.

### 7. What is the use of `annotate()`?

In Django's ORM, `annotate()` is used to attach calculated aggregates or virtual fields to each individual item in a QuerySet. It modifies the underlying SQL query to compute metrics on the database side (such as using `Count`, `Sum`, or `Avg`) and injects the calculated value as an attribute on every object returned (e.g., retrieving a list of blog posts where each post object contains an dynamically calculated `number_of_likes` attribute).

### 8. Is it important to register apps?

Yes, it is mandatory. Every custom app or third-party package must be registered inside the `INSTALLED_APPS` list within your project's `settings.py` file. If an app is not registered, Django will completely ignore its presence—meaning its database models will not generate migrations, its templates will not be recognized, and its signals or administrative commands will fail to execute.

### 9. Difference between `makemigrations` and `migrate`?

* **`makemigrations`**: Inspects your current `models.py` files, compares them against your previous migrations, and drafts a new structural blueprint (stored as a Python file inside the migrations folder) detailing what needs to change. It prepares the instructions but **does not** touch your database tables.
* **`migrate`**: Reads the migration blueprint files that haven't been run yet and executes the actual SQL commands against your database, officially creating, updating, or deleting tables to match your Python code.

### 10. What is `.exclude()` used for?

The `.exclude()` method is a Django QuerySet filter used to omit specific records from your database results. It returns a new QuerySet containing only the rows that do **not** match the given lookup parameters, acting as a logical "NOT" statement in your database query.

### 11. How does `render()` work behind the scenes?

When you call `render(request, template_name, context)`, Django executes a series of automated steps:

1. **Template Loading**: It searches through your configured template directories to find the file matching `template_name`.
2. **Context Binding**: It loads the key-value pairs from your `context` dictionary into memory.
3. **Template Parsing**: The template engine processes the HTML file, replacing template tags (like `{% if %}`) and variable placeholders (like `{{ variable }}`) with actual data values from the context.
4. **Response Wrapper**: It compiles the final fully rendered HTML string into an `HttpResponse` object, sets the content type to `text/html`, assigns the HTTP status code (defaulting to 200 OK), and delivers it back to the web server to be served to the user.
